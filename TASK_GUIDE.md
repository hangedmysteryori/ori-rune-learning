# TASK_GUIDE.md

本文件目的：

在每次交給 Codex 修改網站前，先判斷任務類型、限制修改範圍、指定必要檔案，避免 Codex 每次瀏覽整個專案，減少用量消耗並加速生成結果。

## 1. 核心原則

1. 一次任務只處理一個清楚目標。
2. 先分類任務，再決定允許讀取與修改的檔案。
3. 只讀取完成任務所需的最小範圍。
4. 只修改使用者明確允許的檔案。
5. 檢查與驗收任務不得修改檔案。
6. 小型修改不得擴大成全站重構。
7. 不新增套件、框架、build tool 或 package.json，除非使用者明確要求。
8. 若發現必須修改範圍外檔案，先停止並回報原因。
9. 修改網站前，優先填寫 `CHANGE_BRIEF.md`。
10. 完成後回報修改檔案、修改內容、測試方式與待確認項目。

## 3. 任務分類基準

### A 類：純文案修改

**適用情境**

只調整既有頁面的文字，不改版面、不新增功能、不新增頁面。例如首頁標語、課程文案、占卜說明、FAQ 回答或關於 Ori 的品牌敘述。

**通常需要讀取的檔案**

- 指定頁面的 HTML
- `app.js` 中對應頁面模板
- 文案資料來源（僅在使用者指定時）

**通常允許修改的檔案**

- 指定頁面的 HTML
- `app.js` 中對應的文案

**禁止事項**

- 不改 CSS、功能、路由或其他頁面。
- 不順手重寫未指定文案。
- 若文案同時存在於 `app.js` 與預渲染 HTML，不得只改其中一處。

**指令範例**

> A 類任務：只修改課程頁 Hero 文案。允許讀取與修改 `courses.html`、`app.js`，禁止修改其他檔案與樣式。

### B 類：新增頁面

**適用情境**

新增完整頁面，例如 SEO 文章頁、文章列表、服務頁或活動頁。

**通常需要讀取的檔案**

- 一個既有相似頁面
- `styles.css`
- `app.js`
- `vercel.json`
- `sitemap.xml`
- 導覽或內部連結所在檔案

**通常允許修改的檔案**

- 新增的 HTML
- 必要的 `styles.css`、`app.js`
- `vercel.json`
- `sitemap.xml`
- 明確指定的內部連結來源

**禁止事項**

- 不複製或重寫無關頁面。
- 不在未授權情況下改動全站導覽。
- 不遺漏 title、description、canonical、Open Graph 與正常路由。

**指令範例**

> B 類任務：新增 `/articles/rune-vs-tarot`。只讀取一個既有內容頁作為樣式參考，允許新增文章頁並修改文章列表、sitemap 與必要 rewrite。

### C 類：既有頁面新增區塊

**適用情境**

在既有頁面加入新區塊，但不新增新頁面。例如首頁照片圖文區、占卜流程、課程適合對象或文章入口。

**通常需要讀取的檔案**

- 指定頁面的 HTML
- `app.js` 中對應模板
- `styles.css` 的相關區塊

**通常允許修改的檔案**

- 指定頁面的 HTML
- `app.js`
- `styles.css`
- `assets/` 中明確指定的新素材

**禁止事項**

- 不調整無關頁面與全站結構。
- 不新增外部 UI 套件。
- 新增圖片不得放在專案根目錄，且必須提供 alt。

**指令範例**

> C 類任務：在首頁服務區之前新增一個照片圖文區塊。允許修改 `index.html`、`app.js`、`styles.css`，新增圖片只能放在 `assets/`。

### D 類：CSS 視覺調整

**適用情境**

只調整顏色、按鈕、字距、卡片、照片比例或區塊間距，不修改內容與功能。

**通常需要讀取的檔案**

- `styles.css`
- 必要時讀取指定區塊的 HTML class

**通常允許修改的檔案**

- `styles.css`

**禁止事項**

- 不順手修改 HTML 文案。
- 不修改 JS 功能。
- 不重做全站設計系統。

**指令範例**

> D 類任務：只調整課程卡片的邊框、間距與 hover，允許修改 `styles.css`，禁止修改 HTML 與 JavaScript。

### E 類：RWD 手機版修正

**適用情境**

修正手機、平板或窄螢幕顯示，例如導覽破版、圖片過大、按鈕難點、卡片溢出或表格超出畫面。

**通常需要讀取的檔案**

- `styles.css`
- 指定頁面的 HTML
- 必要時讀取 `app.js` 中產生該區塊的模板

**通常允許修改的檔案**

- `styles.css`
- 只有在 CSS 無法解決時，才修改指定頁面的最小結構

**禁止事項**

- 不改桌機文案與功能。
- 不以隱藏重要內容作為主要修正方式。
- 不擴大成全站改版。

**指令範例**

> E 類任務：修正 390px 寬度下課程 CTA 超出畫面的問題。優先只修改 `styles.css`。

### F 類：JS 互動功能

**適用情境**

調整每日一符文、FAQ、手機選單、按鈕互動、前端路由、頁面 render 或動態 meta。

**通常需要讀取的檔案**

- `app.js`
- 必要時讀取 `data.js`
- 相關 HTML
- 相關 `styles.css`

**通常允許修改的檔案**

- `app.js`
- 功能必要的 `data.js`、指定 HTML 或 `styles.css`

**禁止事項**

- 不因 JS 任務重構全站。
- 不新增框架或套件。
- 不改動無關資料與頁面模板。

**指令範例**

> F 類任務：只修正課程 FAQ 的 aria-expanded 狀態。允許讀取相關 HTML 與 `app.js`，主要只修改 `app.js`。

### G 類：SEO metadata

**適用情境**

修改 title、meta description、canonical、Open Graph、sitemap、robots、自訂網域或 `SITE_ORIGIN`。

**通常需要讀取的檔案**

- 指定頁面的 HTML
- `app.js` 中的 meta 設定
- `sitemap.xml`
- `robots.txt`
- 必要時讀取 `vercel.json`

**通常允許修改的檔案**

- 指定頁面的 HTML
- `app.js`
- `sitemap.xml`
- `robots.txt`
- 必要的 `vercel.json`

**禁止事項**

- 不順手改寫頁面內容與樣式。
- 不在未確認正式網域前批次替換網址。
- 不建立不存在的 og:image 路徑。

**特別檢查**

若更換正式網域，需同步檢查：

- `app.js` 的 `SITE_ORIGIN`
- 各 HTML 的 canonical
- 各 HTML 的 og:url
- 各 HTML 的 og:image
- `sitemap.xml`
- `robots.txt`

若新增 SEO 文章，需同步檢查：

- `articles.html`
- `articles/[slug].html`
- `vercel.json`
- `sitemap.xml`
- 相關內部連結
- title、meta description、canonical
- og:title、og:description、og:url、og:image

**指令範例**

> G 類任務：將正式網域改為指定自訂網域。只修改列出的 SEO 與網域設定，不修改頁面文案或視覺。

### H 類：導覽列與連結修正

**適用情境**

修改站內連結、CTA、導覽列或頁尾連結，例如 LINE、課程 CTA、文章入口或符文詳情連結。

**通常需要讀取的檔案**

- 連結出現的 HTML
- `app.js` 中對應模板或連結常數
- 必要時讀取 `vercel.json`

**通常允許修改的檔案**

- 指定 HTML
- `app.js`
- 必要的 `vercel.json`

**禁止事項**

- 不修改連結周圍的無關文案與樣式。
- 不將站內連結改成不必要的完整網址。
- 外部連結不得遺漏 `target="_blank"` 與 `rel="noopener noreferrer"`。

**指令範例**

> H 類任務：將課程頁所有 LINE CTA 改成指定網址，只修改 `courses.html` 與 `app.js`。

### I 類：檢查與驗收，不修改

**適用情境**

只檢查 SEO、舊網域、盧恩術語、手機版、sitemap 或部署狀態。

**通常需要讀取的檔案**

- 僅讀取檢查目標直接相關的檔案
- 必要時讀取實際頁面或部署結果

**通常允許修改的檔案**

- 無

**禁止事項**

- 禁止修改任何檔案。
- 禁止看到問題後直接修正。
- 禁止擴大掃描範圍，除非使用者要求全站檢查。

**指令範例**

> I 類任務：只檢查 `/courses` 的 SEO 與手機版，不修改檔案。請回報問題、位置與建議。

### J 類：重構與整理

**適用情境**

整理重複 CSS、app.js 函式、SEO route、文章模板或符文資料格式。

**通常需要讀取的檔案**

- 由使用者指定的重構範圍
- 與依賴關係直接相關的檔案

**通常允許修改的檔案**

- 必須在執行前逐項列出
- 只限使用者確認的範圍

**禁止事項**

- 只有使用者明確要求時才可執行。
- 執行前必須列出影響範圍與風險。
- 不得自行將小任務擴大成重構。
- 不得以「順便整理」為由修改無關檔案。

**指令範例**

> J 類任務：只整理 `app.js` 中重複的 SEO meta 函式。先列出影響範圍與風險，獲得確認後再修改。

## 4. 修改前說明書模板

每次修改前，請先複製並填寫 `CHANGE_BRIEF.md`，至少確認：

- 任務名稱與 A～J 類型
- 單一任務目標
- 允許讀取的檔案
- 允許修改的檔案
- 禁止修改範圍
- 文案、功能或素材內容
- SEO、網域與部署需求
- 完成標準與測試方式

若任務只做檢查，任務類型應選 I 類，並將「允許修改的檔案」填為「無」。

## 5. Codex 任務前快速判斷表

| 問題 | 判斷方式 |
| --- | --- |
| 只是改文字嗎？ | A 類 |
| 要建立新網址或新 HTML 嗎？ | B 類 |
| 既有頁面增加一個區塊嗎？ | C 類 |
| 只改顏色、間距、卡片或字體嗎？ | D 類 |
| 只修手機或平板破版嗎？ | E 類 |
| 涉及點擊、抽取、選單、accordion 或 route 嗎？ | F 類 |
| 涉及 title、canonical、OG、sitemap 或網域嗎？ | G 類 |
| 只改 CTA、導覽或連結嗎？ | H 類 |
| 只要報告，不允許修正嗎？ | I 類 |
| 要整理架構、移除重複或改資料格式嗎？ | J 類 |

若同時符合多類，請選一個主要類型，並在說明書中列出次要類型與必要檔案，不要因此開放全站修改。

## 6. 建議使用流程

1. 明確說出本次只要完成的一件事。
2. 使用快速判斷表選擇 A～J 類。
3. 複製並填寫 `CHANGE_BRIEF.md`。
4. 列出允許讀取與修改的檔案。
5. 將正式文案、網址或圖片路徑一併提供。
6. 要求 Codex 在修改前回報範圍衝突。
7. 修改後依完成標準測試。
8. 確認無範圍外修改後再部署。

高效率句型應包含：

- 任務類型
- 單一目標

## 8. 最不建議的任務句型

以下句型範圍不清，容易造成過度掃描、用量增加或誤改：

> 幫我優化整個網站。

> 幫我看看哪裡可以更好，順便全部修掉。

> 幫我做 SEO、改設計、整理程式並新增幾個頁面。

> 有問題就直接改，不用問我。

若需求較大，應拆成多個 A～J 類任務，逐次完成與驗收。

## 9. 結論

每次任務都應先分類、再限制檔案範圍，最後才執行修改。

最重要的原則是：

- 小任務只讀小範圍。
- 只修改必要檔案。
- 檢查任務不修改。
- 重構必須明確授權。
- 網站內容、互動、SEO 與部署設定不可在未授權時互相擴張。
